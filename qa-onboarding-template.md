# QA Onboarding: Старт на новом проекте

Шаблон для первого QA на проекте без тестов и процессов.
Ориентирован на AI-first продукты (генерация, async pipelines, LLM-интеграции).

---

## Принципы

- **Риск прежде всего.** Тестируем то, что критично для бизнеса и пользователей — не всё подряд.
- **Свежий взгляд — невозобновляемый ресурс.** Исследовательское тестирование начинается в первые дни. Через месяц это уже не сделать.
- **Документация живёт только если её легко обновить.** Одна актуальная страница ценнее десяти устаревших. Короткие живые документы лучше подробных мёртвых.
- **Процесс важнее покрытия.** Регулярный регресс с 50 кейсами ценнее 500 кейсов без процесса.

---

## Фаза 0 — Онбординг (день 1–3)

### Доступы — запросить в первый день

- [ ] Staging / dev окружение + тестовые аккаунты (разные тарифы и роли)
- [ ] Репозиторий
- [ ] Jira
- [ ] TMS (или договориться о выборе — Qase, TestRail и т.п., не Jira)
- [ ] Swagger / API docs
- [ ] Notion / Confluence
- [ ] CI/CD — хотя бы read-only
- [ ] Логи и мониторинг (Sentry, Datadog, etc.)

### Разговор с продактом — бизнес-риски

- [ ] Что самое важное для пользователя? Без чего он уйдёт?
- [ ] Какие фичи используются больше всего?
- [ ] На что жалуются пользователи прямо сейчас?
- [ ] Что нельзя сломать ни при каких обстоятельствах?
- [ ] Что планируется к релизу в ближайшие 1–2 месяца?

### Разговор с разработчиками — технические риски

- [ ] Что чаще всего ломается?
- [ ] Чего боитесь трогать в коде?
- [ ] Какие части кода менялись последние 2–3 месяца? (= зоны нестабильности)
- [ ] Есть ли известные баги, которые пока не фиксят?
- [ ] Какие внешние зависимости критичны и насколько они стабильны?
- [ ] Как сейчас выглядит релизный процесс?

> **AI-first:** Спросить про LLM-провайдеров: кто используется, были ли проблемы с их доступностью или изменением API, как обрабатываются таймауты и ошибки провайдера.

### Теневые QA-процессы

До появления QA кто-то всё равно проверял — неформально.

- [ ] Кто кликает по продукту перед деплоем? Как это происходит?
- [ ] Есть ли неформальные чеклисты или "надо не забыть проверить X"?
- [ ] Кто идёт в первую очередь когда что-то сломалось в проде?
- [ ] Как сейчас принимается решение "можно деплоить"?

> Эти процессы существуют почти всегда. Их важно найти и не сломать — они держат продукт живым пока ты настраиваешь нормальный процесс.

→ **Выход фазы:** список "больных мест" с двух сторон (бизнес + техника) + понимание теневых процессов.

## Фаза 1 — Разведка (неделя 1)

### СРАЗУ: исследовательское тестирование (день 1–2)

Начинать параллельно с онбордингом, не ждать полного понимания продукта.

- [ ] Пройти продукт глазами нового пользователя — без плана, без сценариев
- [ ] Фиксировать всё непонятное, неудобное, сломанное без фильтрации
- [ ] Отмечать UX-проблемы отдельно: не только "не работает", но и "работает, но непонятно" или "работает, но неудобно"
- [ ] Записывать вопросы "почему так сделано?" — часть из них окажется багами

Первые баг-репорты появятся здесь же. Не ждать правильного процесса — фиксировать сразу.

> **AI-first:** Отдельно пройти полный цикл генерации: отправить задачу, дождаться результата, скачать. Зафиксировать: сколько времени занимает, понятен ли статус ожидания пользователю, что происходит при ошибке генерации.

### Анализ репозитория

- [ ] Понять структуру: модули, сервисы, основные сущности и их связи
- [ ] Найти самые изменяемые файлы (`git log --stat`) — зоны нестабильности
- [ ] Найти внешние интеграции: платёжка, email, авторизация
- [ ] Посмотреть CI/CD конфиг: что сейчас проверяется автоматически

> **AI-first:** Найти и понять: очереди задач (Redis, RabbitMQ, SQS и т.п.), воркеры генерации, интеграции с LLM-провайдерами, логику retry и обработки ошибок провайдера, хранение и отдачу медиафайлов.

### Анализ Swagger / OpenAPI

- [ ] Выписать все эндпоинты по группам (auth, billing, core features)
- [ ] Отметить критичные — без которых продукт не работает
- [ ] Найти неочевидные или слабо задокументированные поля

> **AI-first:** Отдельно картировать async-эндпоинты: сабмит задачи, проверка статуса, получение результата, вебхуки. Зафиксировать все возможные статусы джоба и переходы между ними.

### Анализ Notion / Confluence

- [ ] Найти описание основных user flows
- [ ] Найти известные ограничения и edge cases
- [ ] Проверить актуальность документов — если старые, флагировать

→ **Выход фазы:** карта рисков.

**Формат карты рисков:**

| Фича / область | Риск | Приоритет | Тип тестирования |
|----------------|------|-----------|-----------------|
| Авторизация | Нет доступа к продукту | High | Ручное + авто |
| Генерация изображений | Core feature не работает | Critical | Ручное + авто |
| Биллинг | Потеря оплаты | Critical | Ручное |
| Лимиты плана | Бесплатный доступ к платным фичам | High | Ручное + авто |
| ... | ... | ... | ... |

## Фаза 2 — Документация (неделя 1–2)

### Принцип поддерживаемости

> Документ, который сложно обновить, перестанет обновляться. Писать минимум необходимого. Обновлять при каждом изменении фичи. Устаревшее — удалять или архивировать, не оставлять.

### Запросить у команды

- [ ] Описание user flows для ключевых фич (если нет — составить самому по итогам разведки и согласовать)
- [ ] Описание релизного процесса: как деплоят, как часто, кто принимает решение
- [ ] Что уже мониторится: алерты, дашборды, on-call

> **AI-first:** Согласовать технические критерии приёмки генерации — это обязательно до написания тестов. Договориться что такое "успешный результат": форматы файлов, допустимые размеры, максимальное время генерации, коды ошибок и их смысл. Без этого ручное тестирование субъективно, автоматизация невозможна.

### Создать самому

- [ ] **Тест-стратегия** (1–2 страницы): что тестируем, что не тестируем и почему, подход к приоритизации рисков
- [ ] **Карта рисков** (таблица из фазы 1): фича → риск → приоритет → тип тестирования
- [ ] **Severity guide**: единые правила классификации багов для команды (Critical / High / Medium / Low с примерами)
- [ ] **Гайд по тест-окружению**: как получить доступ, какие аккаунты использовать, особенности staging

### Выбрать TMS

- [ ] Договориться о системе для тест-кейсов (Qase, TestRail, PractiTest и т.п.) — не Jira
- [ ] Настроить структуру разделов по фичам / модулям
- [ ] Настроить связь тест-кейсов с задачами в Jira

## Фаза 3 — Тест-кейсы (неделя 2–3)

### Подход

- Писать по карте рисков: Critical → High → Medium
- На каждую фичу: happy path → негативные сценарии → edge cases
- Кейс длиннее 10 шагов — разбить на несколько

### Структура кейса

- **Предусловие:** состояние системы и пользователя перед тестом
- **Шаги:** конкретные, однозначные, воспроизводимые другим человеком
- **Ожидаемый результат:** проверяемый, без "должно работать нормально"
- **Postcondition:** если нужно вернуть систему в исходное состояние

> **AI-first:** Для генерации — отдельные кейсы на: успешный цикл (submit → polling → result), таймаут генерации, ошибку провайдера, невалидный промпт, скачивание результата, повторный запрос после ошибки. Ожидаемый результат формулировать через технические критерии (из фазы 2), не субъективные оценки.

### Приоритет написания

1. Critical и High flows из карты рисков
2. Релизный регресс-набор (что проверяем перед каждым деплоем)
3. Medium риски
4. Edge cases по мере нахождения

### Поддерживаемость тест-кейсов

- При изменении фичи — сразу обновить связанные кейсы
- Устаревшие кейсы архивировать, не копить мусор
- Не детализировать UI-шаги избыточно — кнопки переезжают, логика остаётся

## Фаза 4 — Регресс (неделя 3)

### Определить scope

- [ ] Выделить в TMS набор "релизный регресс" — Critical + High кейсы
- [ ] Замерить время полного прогона (ориентир для ручного: ≤ 4 часов)
- [ ] Зафиксировать два режима: полный регресс (релиз) и сокращённый (хотфикс)

### Релизный чеклист

- [ ] Создать чеклист в Confluence / Notion: что проверить перед деплоем на прод
- [ ] Договориться с командой: QA sign-off обязателен перед релизом
- [ ] Фиксировать результат каждого прогона: дата, версия, passed/failed, найденные баги

> **AI-first:** В релизный чеклист включить отдельной строкой: полный цикл генерации прошёл успешно на staging с реальным провайдером.

## Фаза 5 — Автоматизация (неделя 3–4)

- [ ] Настроить фреймворк под стек проекта
- [ ] **Smoke suite первым:** 5–10 тестов на самые критичные пути, прогон за < 5 минут
- [ ] Подключить к CI/CD: smoke запускается на каждый деплой в staging
- [ ] Расширять автоматизацию только на устоявшийся функционал

> Автоматизация активно меняющегося кода = постоянная поддержка сломанных тестов. Сначала фича стабилизировалась — потом автоматизация.

> **AI-first:** Мокировать LLM-провайдеров на уровне HTTP-транспорта (не в бизнес-логике) — реальные вызовы в CI дорогие и нестабильные. Написать хелперы для async-паттерна: `submit_job → poll_status → assert_result`. Реального провайдера подключать только в ручном тестировании и нагрузочных тестах на staging.

---

## Фаза 6 — Рост (месяц 2+)

- [ ] Расширять автоматизацию по карте рисков: API тесты → E2E golden paths
- [ ] Добавить нагрузочные тесты для критичных async flows
- [ ] Настроить метрики: количество тестов, flakiness rate, время прогона
- [ ] Ретроспектива с командой: что QA улучшило, что ещё болит, что менять в процессе

> **AI-first:** Нагрузочные тесты — в первую очередь на job queue: параллельные сабмиты, saturation point очереди, деградация времени генерации под нагрузкой.

---

## Быстрый старт — первые 3 дня

Если надо начать немедленно и план кажется большим:

1. Запросить доступы
2. Поговорить с продактом и одним разработчиком
3. Начать трогать продукт руками — фиксировать всё
4. Выписать топ-5 самых страшных сценариев поломки

Остальное придёт из этого.
